# Plan: `test/massiveOnchainTest.ts` — Multi-Pair Escrow Bot

Membuat bot onchain baru yang menjalankan siklus escrow ERC-8183 menggunakan **1000 wallet bot dari `wallet.json`** yang dibiayai oleh **2 wallet funding dari `.env`**. Alur inti sama dengan `onchainTest.ts`, tapi diadaptasi untuk pemrosesan ber-pair.

## Spesifikasi yang sudah dikonfirmasi

- **Sumber dana:** 2 wallet funding dari `.env` (`FUNDING_WALLET_PRIVATE_KEY_1`, `FUNDING_WALLET_PRIVATE_KEY_2`)
- **Wallet bot:** 1000 wallet dari `wallet.json`, dipasangkan **fixed**: Pair 1 = idx (0,1), Pair 2 = (2,3), ..., Pair 500 = (998,999). Wallet genap (idx 0,2,4,...) = **Client**; wallet ganjil (idx 1,3,5,...) = **Worker**
- **Threshold per cycle:** Client ≥ 5 USDC, Worker ≥ 1 USDC
- **Topup saat trigger:** Client <5 → funding kirim **+10 USDC**; Worker <1 → funding kirim **+5 USDC**
- **Budget job:** acak **1–4 USDC** per cycle (diambil dari balance client)
- **Sweep balance:** sisa balance pair N → dipindahkan ke pair N+1 sebelum pair N+1 mulai
- **Loop:** abadi (500 pair → kembali ke pair 1-2 → ulang)
- **DB sync:** setiap cycle sync ke Supabase (buat/take profile + insert job + update statistik)
- **Final cleanup (saat exit via Ctrl+C):** sweep semua sisa balance 1000 wallet bot kembali ke funding wallet

## Pertimbangan teknis

### 1. Manajemen rate-limit RPC (KRITIKAL)
Bot ini memproses 1000 wallet × multiple RPC call per cycle = ribuan request. Solusi:
- Reuse `rateLimitedTransport` dengan gap dinamis (mulai 500ms, naik saat kena -32011)
- `STAGE_DELAY` antar step (5 detik, sama dgn onchainTest.ts)
- Gap antar pair (mis. 3 detik) untuk breathing room
- Semua client berbagi satu `throttleQueue` global

### 2. Dynamic wallet client
Tidak bisa pre-create 1000 `walletClient` di memory. Buat lazily:
```
function walletFrom(privateKey) {
  const account = privateKeyToAccount(privateKey);
  return createWalletClient({ account, chain: arcTestnet, transport: rateLimitedTransport });
}
```
Lebih bagus: cache di Map<address, WalletClient> supaya tidak re-create saat loop abadi.

### 3. Funding wallet selection
2 funding wallet untuk topup paralel (funding1 → client topup, funding2 → worker topup). Keduanya harus punya USDC testnet yang cukup untuk 500 pair. Funding wallet 1 bisa = feeRecipient untuk recover 1% platform fee (opsional, tidak di-hardcode).

### 4. Budget randomisasi
```
function randomBudget() {
  const min = 1_000000;  // 1 USDC
  const max = 4_000000;  // 4 USDC
  return BigInt(randomBetween(1, 4)) * 1_000000n;
}
```

### 5. DB sync consistency
`syncJobToDatabase()` membuat profile baru per wallet address. 1000 wallet → 1000 profile. Ini OK sesuai konfirmasi user ("Ya, sync semua"). Pakai `maybeSingle` + insert-or-skip pattern yang sudah ada.

### 6. Error handling per pair
- Rate limit cooldown (60s) → continue ke pair berikutnya
- Pair gagal non-rate-limit → log + skip, sweep apa adanya ke pair berikutnya
- `recoverStuckJobs()` di awal tiap iterasi loop abadi untuk safety net
